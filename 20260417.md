# Aula: Tratamento de Exceções em JavaScript (Síncrono)

## Parte 1: A Teoria (Metade da Aula)
*O "Porquê", a Engenharia de Software e a Prática no Código.*

### 1. O que são Exceções e por que usá-las?
Até agora, os alunos aprenderam a programar o "caminho feliz". Mas o mundo real é caótico: usuários digitam letras no lugar de números, e dados chegam corrompidos. 

**O que é uma exceção?** É um evento anômalo que interrompe o fluxo normal do programa. Quando o JavaScript encontra um erro crítico, ele "lança" (throw) uma exceção. Se ninguém "capturar" (catch) essa exceção, o programa quebra (a temida tela branca ou o script que para de rodar).

**Por que não usar apenas `if/else`?**
Historicamente, programadores retornavam "códigos de erro" (ex: retornar `-1` se falhasse). Isso obrigava o código a ter dezenas de `if`s espalhados para verificar o retorno de cada função. O código ficava ilegível e os erros eram facilmente ignorados.

### 2. A Visão da Engenharia de Software
A Engenharia de Software moderna exige o uso de exceções por três motivos:

1.  **Separação de Preocupações (Separation of Concerns):** Exceções separam a "lógica de negócio" da "lógica de tratamento de erros". Você lê o que o programa deve fazer em um bloco de código, e como ele lida com tragédias em outro.
2.  **Fail-Fast (Falhe Rápido):** É melhor o programa quebrar e avisar imediatamente o que deu errado do que continuar rodando com um dado corrompido, o que poderia salvar informações erradas no banco de dados.
3.  **Rastreabilidade (Stack Trace):** Uma exceção real (`new Error()`) carrega consigo a "Pilha de Chamadas". Ela diz ao programador exatamente em qual linha, de qual arquivo, a função falhou. Isso separa o programador amador do profissional na hora do *debugging*.

### 3. A Sintaxe Base: O Mundo Real com `JSON.parse`
Um dos melhores exemplos síncronos de exceção no dia a dia da web é a conversão de textos para objetos JSON.

```javascript
const dadosCorrompidos = "{ nome: 'João', idade: 25 "; // Falta fechar a chave!

try {
  console.log("1. Tentando ler os dados...");
  
  // O JSON.parse nativamente lança uma exceção se a string for inválida
  const usuario = JSON.parse(dadosCorrompidos); 
  
  console.log("2. Isso não será executado se houver erro!", usuario);

} catch (erro) {
  // O JavaScript pula para cá imediatamente na falha
  console.error("Ops! Os dados vieram corrompidos.");
  console.error("Detalhes técnicos:", erro.message); // Exibe o motivo exato
} finally {
  // Executa SEMPRE, dando certo ou errado.
  console.log("3. Operação de leitura finalizada. Limpando memória temporária...");
}
```

### 4. Lançando seus próprios erros (`throw`)
Nós não precisamos esperar o JavaScript dar erro. Nós devemos lançar erros quando as **regras do nosso negócio** são violadas.

> **Regra de Ouro:** Sempre lance objetos `new Error()`, nunca jogue strings soltas (`throw "Erro"`), para não perder o Stack Trace.

```javascript
function sacarDinheiro(saldo, valorSaque) {
  if (typeof valorSaque !== 'number') {
    throw new TypeError("O valor do saque deve ser um número.");
  }
  if (valorSaque > saldo) {
    throw new Error("Saldo insuficiente para esta operação.");
  }
  return saldo - valorSaque;
}

try {
  let novoSaldo = sacarDinheiro(100, "cinquenta"); // Vai disparar o TypeError
} catch (erro) {
  console.warn("Transação negada:", erro.message);
}
```

### 5. Classes de Erro Personalizadas
Para sistemas complexos, criamos nossas próprias categorias de erro. Isso permite que o `catch` saiba exatamente o que deu errado.

```javascript
class RegraDeNegocioError extends Error {
  constructor(mensagem) {
    super(mensagem);
    this.name = "RegraDeNegocioError";
  }
}

function cadastrarSenha(senha) {
  if (senha.length < 6) {
    throw new RegraDeNegocioError("A senha deve ter no mínimo 6 caracteres.");
  }
  console.log("Senha cadastrada com sucesso!");
}
```

## Parte 2: O Laboratório (Outra Metade da Aula)
*Mão na massa. O objetivo é criar um sistema puramente lógico e síncrono que exija o controle rigoroso de fluxo.*

### O Cenário
Você está desenvolvendo o núcleo (core) do sistema de um **Caixa de Supermercado**. O sistema precisa registrar os produtos, aplicar descontos e finalizar a compra. Há muitas regras rigorosas, e qualquer violação deve bloquear a venda e avisar o operador do caixa imediatamente.

### Objetivos do Laboratório
1. Criar erros personalizados para separar falhas de sistema de erros de operação.
2. Usar `throw` para garantir as regras de negócio de forma "fail-fast".
3. Controlar o estado da interface do caixa usando o bloco `finally`.

### Passo a Passo

**Passo 1: Preparando as Exceções**
Crie duas classes que herdam de `Error`:
* `CodigoInvalidoError`: Para quando o código de barras for inválido.
* `EstoqueVazioError`: Para quando tentar vender um produto sem estoque.

**Passo 2: O Banco de Dados Simulado (Array)**
Crie um array simples de produtos fora de qualquer função:
```javascript
const produtos = [
    { codigo: 101, nome: "Arroz", preco: 20.00, estoque: 5 },
    { codigo: 102, nome: "Feijão", preco: 10.00, estoque: 0 }
];
```

**Passo 3: A Função de Busca (Regras de Negócio)**
Crie a função `buscarProduto(codigoBuscado)`.
* Tente encontrar o produto no array (use `.find()`).
* Se não encontrar o produto, **lance** um `CodigoInvalidoError` com a mensagem: *"Produto não cadastrado no sistema."*
* Se encontrar, mas o `estoque` for igual a `0`, **lance** um `EstoqueVazioError` com a mensagem: *"Produto fora de estoque. Chame o supervisor."*
* Se der tudo certo, retorne o objeto do produto.

### Exemplo
```

const alunos = [
    { nome: "carlos", curso: "Engenharia", notaFinal: 8.5 },
    { nome: "ana", curso: "Computação", notaFinal: 9.0 },
    { nome: "pedro", curso: "Administração", notaFinal: 4.5 }
];

// A função find procura o aluno que tem o nome igual a "carlos"
const resultado = alunos.find(aluno => aluno.nome === "carlos");

if (!resultado) {
        // Lança o erro específico de código inválido e PARA a função aqui
        throw new CodigoInvalidoError("Aluno não cadastrado no sistema.");
}

console.log(resultado); 
// Imprime o objeto inteiro: { nome: 'carlos', curso: 'Engenharia', notaFinal: 8.5 }

```

**Passo 4: A Função do Caixa (O Controle de Fluxo)**
Crie a função `registrarNoCaixa(codigo)`. Ela deve simular a ação do operador do supermercado passando o leitor de código de barras.
1.  Crie uma variável `let caixaOcupado = true;` e imprima `"Lendo código de barras..."`.
2.  Abra um bloco `try`.
3.  Chame a função `buscarProduto(codigo)` e guarde o retorno em uma variável.
4.  Imprima: `"BIP! [Nome do Produto] registrado. Valor: R$ [Preço]"`.
5.  Abra o bloco `catch (erro)`.
    * *Desafio extra para os alunos:* Usem `if (erro instanceof EstoqueVazioError)` para imprimir um aviso em amarelo (`console.warn`) e um `else` genérico para imprimir um erro em vermelho (`console.error`).
6.  Abra o bloco `finally`.
    * Mude `caixaOcupado = false;` e imprima `"Caixa liberado para o próximo item."`

### 💡 Dicas do Professor para o Quadro

* **Por que não retornar `false` se não achar o produto?** Porque se retornarmos `false`, a função do caixa tentará ler `false.nome` e o JavaScript vai explodir com um erro genérico incompreensível. A exceção nos dá o controle da narrativa do erro!
* Lembrem-se que o `finally` é o lugar perfeito para destravar botões na tela (ou no nosso caso, liberar a variável `caixaOcupado`), garantindo que o sistema nunca fique "travado" esperando algo após um erro.



### Código espaguete

```
// 1. Função mais profunda (simula busca no banco)
function buscarUsuarioNoBanco(id) {
    if (id < 1) {
        // Retorna um objeto indicando o erro
        return { sucesso: false, erro: "ID de usuário inválido." };
    }
    return { sucesso: true, dados: { id: id, nome: "Maria" } };
}

// 2. Função intermediária (simula validação)
function validarUsuario(id) {
    const resultadoBusca = buscarUsuarioNoBanco(id);
    
    // Checagem obrigatória: se deu erro, repassa o erro para cima
    if (!resultadoBusca.sucesso) {
        return { sucesso: false, erro: "Falha na validação: " + resultadoBusca.erro };
    }
    
    // Caminho feliz
    const usuario = resultadoBusca.dados;
    return { sucesso: true, dados: usuario };
}

// 3. Função principal (simula o controlador)
function iniciarProcesso(id) {
    const resultadoValidacao = validarUsuario(id);
    
    // Checagem final para decidir o que fazer
    if (!resultadoValidacao.sucesso) {
        console.error("❌ Erro ao processar:", resultadoValidacao.erro);
        return;
    }
    
    console.log("✅ Sucesso! Usuário encontrado:", resultadoValidacao.dados.nome);
}

// Testando
iniciarProcesso(-5); // Retorna erro repassado

```

### Código limpo

```
// 1. Função mais profunda
function buscarUsuarioNoBanco(id) {
    if (id < 1) {
        // Lança uma exceção (interrompe a execução aqui mesmo)
        throw new Error("ID de usuário inválido.");
    }
    return { id: id, nome: "Maria" };
}

// 2. Função intermediária
function validarUsuario(id) {
    // Código limpo! Sem "ifs" de checagem de erro.
    // Se 'buscarUsuarioNoBanco' falhar, a exceção passa direto por aqui.
    const usuario = buscarUsuarioNoBanco(id);
    
    return usuario;
}

// 3. Função principal (Apenas aqui tratamos o erro)
function iniciarProcesso(id) {
    try {
        // Bloco de fácil compreensão: tentamos executar o "caminho feliz"
        const usuario = validarUsuario(id);
        console.log("✅ Sucesso! Usuário encontrado:", usuario.nome);
        
    } catch (erro) {
        // Se QUALQUER função dentro do 'try' lançar um erro, a execução
        // pula direto para cá.
        console.error("❌ Erro ao processar:", erro.message);
    }
}

// Testando
iniciarProcesso(-5); // O erro viaja direto do 'buscarUsuarioNoBanco' para o 'catch'
```
