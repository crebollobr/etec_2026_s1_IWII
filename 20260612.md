# Parte I — Teoria

## 1. Mudança importante: vamos sair do JSONPlaceholder

Até agora a gente usou a **JSONPlaceholder**. Ela aceita até POST (devolve status 201, devolve um id novo), mas **finge** — não salva nada. Você cria uma tarefa, atualiza a página, e a tarefa sumiu.

Pra aprender o que **realmente acontece** quando você envia dados, o professor montou um servidor próprio em **`https://curso.chr.eti.br/ajax/api.php`** — um único arquivo PHP que guarda os dados em um banco **SQLite**.

A partir desta aula, você vai criar registros, eles vão ser salvos de verdade, e quando você abrir o navegador amanhã, eles vão estar lá.

## 2. A API do curso

URL base: **`https://curso.chr.eti.br/ajax/api.php`**

Recurso único: **tarefas**. Cada tarefa tem:

| Campo        | Tipo     | O que é                                                 |
|--------------|----------|---------------------------------------------------------|
| `id`         | número   | identificador único, gerado pelo servidor               |
| `aluno_id`   | texto    | quem é o dono — você escolhe a sua string               |
| `titulo`     | texto    | o que a tarefa é                                        |
| `feito`      | 0 ou 1   | se já está concluída                                    |
| `criado_em`  | texto    | data/hora de criação, automático                        |

Os endpoints (vamos usar todos nas próximas 3 aulas):

| O que quer fazer            | Como                                                       |
|-----------------------------|------------------------------------------------------------|
| Listar suas tarefas         | `GET    api.php?aluno_id=ana123`                           |
| Pegar uma tarefa            | `GET    api.php?aluno_id=ana123&id=7`                      |
| **Criar tarefa (hoje)**     | `POST   api.php` + corpo `{aluno_id, titulo}`              |
| Atualizar (aula 5)          | `PUT    api.php?id=7` + corpo JSON                         |
| Apagar (aula 5)             | `DELETE api.php?id=7&aluno_id=ana123`                      |

## 3. O `aluno_id` — por que precisa

A **turma inteira** vai usar o mesmo servidor. Se não tivesse uma forma de separar quem criou cada tarefa, você ia ver as tarefas dos seus colegas misturadas com as suas.

O servidor **filtra tudo por `aluno_id`**. Você escolhe a sua identificação (uma string qualquer), envia em toda chamada, e a API só te devolve / aceita coisas com aquele id.

**Não é login, não é segurança.** É **organização**. Se outro aluno digitar seu id, ele vê suas tarefas. Tudo bem — é ambiente didático.

Sugestão de formato: **`primeironome + 3 dígitos`** (ex: `ana123`, `bruno007`, `gustavo542`). Escolha **uma vez**, anote, e use sempre o mesmo nas próximas 3 aulas.

## 4. `localStorage` — guardar coisas no navegador

Você vai usar seu `aluno_id` em **todas** as chamadas dessa aula e das próximas. Em vez de digitar toda vez (ou hardcodar no JS), a gente guarda **uma vez** no `localStorage` e lê depois.

`localStorage` é uma caixinha de chave-valor que **persiste mesmo depois de fechar o navegador**. Você já viu em JS puro:

```javascript
localStorage.setItem('aluno_id', 'ana123');    // grava
var id = localStorage.getItem('aluno_id');     // lê → 'ana123'
localStorage.removeItem('aluno_id');           // apaga
localStorage.clear();                           // apaga TUDO
```

Características:

- Só guarda **strings**. Pra guardar objeto, você converte: `JSON.stringify(obj)` para gravar e `JSON.parse(str)` para ler.
- É **separado por site** — `curso.chr.eti.br` não vê o que `google.com` guardou.
- Tem limite de ~5MB por site. Mais que suficiente pra qualquer coisa normal.
- Persiste até o usuário apagar manualmente (em "Limpar dados do navegador").

## 5. POST com `$.post`

Pra **enviar** dados ao servidor, use `$.post`:

```javascript
$.post(url, dados, function(resposta) {
    // resposta = o que o servidor devolveu
}, 'json');
```

Parâmetros:

- `url` — endereço.
- `dados` — objeto JS com os campos. jQuery transforma em `chave=valor&chave=valor` (formato "form URL-encoded", o padrão de formulário HTML).
- `função` — callback. Chama quando a resposta chega.
- `'json'` — diz pra esperar resposta em JSON (jQuery já faz o `JSON.parse` automaticamente).

**Exemplo concreto** — criar uma tarefa:

```javascript
var dados = {
    aluno_id: 'ana123',
    titulo: 'Estudar AJAX'
};

$.post('https://curso.chr.eti.br/ajax/api.php', dados, function(tarefa) {
    console.log('Criada com id', tarefa.id);
}, 'json');
```

A `tarefa` que volta é a tarefa **completa** salva no banco — com `id` novo, `aluno_id`, `titulo`, `feito: 0`, `criado_em` preenchido.

## 6. O que viaja na rede

Quando você faz esse POST acima, isso é o que sai do navegador (você pode ver na aba **Network** do F12):

```
POST /ajax/api.php HTTP/1.1
Host: curso.chr.eti.br
Content-Type: application/x-www-form-urlencoded

aluno_id=ana123&titulo=Estudar+AJAX
```

E isso é o que volta:

```
HTTP/1.1 201 Created
Content-Type: application/json

{
    "id": 42,
    "aluno_id": "ana123",
    "titulo": "Estudar AJAX",
    "feito": 0,
    "criado_em": "2026-06-12 19:42:08"
}
```

**Sempre** abra a Network nos exercícios pra ver isso acontecer. É a coisa mais importante da aula.

## 7. Lendo formulários: `.val()` campo a campo vs `.serialize()` o form todo

Pra um form pequeno:

```html
<form id="meuForm">
    <input id="titulo" name="titulo">
    <input id="prazo" name="prazo" type="date">
</form>
```

Você pode pegar campo a campo:

```javascript
var dados = {
    titulo: $('#titulo').val(),
    prazo:  $('#prazo').val()
};
```

Ou pegar **o form inteiro de uma vez**, com `.serialize()`:

```javascript
var stringPronta = $('#meuForm').serialize();
// "titulo=Estudar+AJAX&prazo=2026-06-15"
```

`.serialize()` usa o atributo `name` de cada campo. Se faltar o `name`, o campo é **ignorado**. Cuidado.

Você pode passar essa string direto pro `$.post`:

```javascript
$.post(url, $('#meuForm').serialize(), callback, 'json');
```

## 8. `e.preventDefault()` — evitando o reload

Formulário HTML, **por padrão**, ao ser submetido, **recarrega a página**. Isso destrói tudo que seu AJAX fez.

Pra impedir esse comportamento padrão, capture o evento `submit` e chame `e.preventDefault()`:

```javascript
$('#meuForm').on('submit', function(e) {
    e.preventDefault();   // segura o reload
    // agora faz o AJAX
});
```

Por que usar `submit` em vez de `click` no botão? Porque `submit` cobre **mais cenários**: clique no botão, Enter no campo, submit programático. Sempre prefira `submit` em formulário.

## 9. Padrão "criar e recarregar"

Quando o aluno cria algo, ele espera **ver na tela** logo em seguida. O padrão é:

```
1. usuário clica em "Adicionar"
2. POST envia os dados
3. quando volta sucesso → recarrega a lista (GET)
4. lista atualizada renderiza, mostrando o item novo
```

Em código:

```javascript
function carregarLista() {
    $.getJSON(url, function(itens) { /* renderiza */ });
}

$('#form').on('submit', function(e) {
    e.preventDefault();
    $.post(url, $(this).serialize(), function() {
        $('#form input').val('');   // limpa o form
        carregarLista();             // recarrega
    }, 'json');
});

carregarLista();   // primeira carga
```

Existe uma alternativa **otimista**: você adiciona o novo item na lista **antes** da resposta chegar (porque "vai dar certo"), só corrige se falhar. É mais rápido visualmente, mas mais complicado de programar. Vamos usar o padrão "criar e recarregar" por enquanto — é mais simples e funciona bem.

## 10. Validação visual antes de enviar

Quanto mais cedo você impedir um envio inútil, melhor. Regras típicas:

- Campo obrigatório vazio? Não envia.
- Tamanho mínimo / máximo? Confere.
- Formato (email com `@`, idade entre 1 e 120, etc.)? Confere.

**Padrão visual padrão da web:**

- Borda do campo fica **vermelha**.
- Mensagem de erro aparece **perto** do campo (não num alert).
- Quando o usuário **volta a digitar**, o erro **some** — não fica vermelho até ele clicar de novo em "Enviar".

```css
.campo input { border: 2px solid #ccc; }
.campo.erro input { border-color: crimson; background: #fff0f0; }
.campo .msg { display: none; color: crimson; }
.campo.erro .msg { display: block; }
```

```javascript
// quando der erro:
$('#campoTitulo').addClass('erro');
$('#msgTitulo').text('Mínimo 3 caracteres.');

// quando o usuário digita, limpa:
$('#titulo').on('input', function() {
    $('#campoTitulo').removeClass('erro');
});
```

## 11. `encodeURIComponent` — quando precisa

Se o seu `aluno_id` (ou qualquer dado) tem caractere especial — espaço, `&`, `=`, `+`, acento — colocando direto na URL ele **quebra**:

```javascript
// errado:
var url = 'api.php?aluno_id=' + alunoId;
// se alunoId for "ana e bruno", a URL vira ?aluno_id=ana e bruno (quebrado)
```

Use `encodeURIComponent` pra escapar:

```javascript
var url = 'api.php?aluno_id=' + encodeURIComponent(alunoId);
// "ana e bruno" vira "ana%20e%20bruno" — válido
```

Como regra: **sempre encode** o que você bota em query string.

## Exercício 5 — Recados com prioridade

Crie **`lab23/ex05.html`** parecido com o ex04, mas o form tem **dois campos** que se juntam num único `titulo`.

# Parte III — 🏆 Desafio

Crie **`lab23/desafio.html`** — **agenda de contatos**.

A API só conhece "tarefas", mas você vai **usar o campo `titulo` criativamente** pra guardar 3 campos de um contato concatenados:

```
"Pedro Silva | pedro@email.com | (11) 99999-0000"
```

E na hora de mostrar, **separa de volta** com `.split(' | ')`.

**Visual sugerido:**

```
+----------------------------------+
| Agenda — 3 contatos              |
+----------------------------------+
| Nome:    [_________________]     |
| Email:   [_________________]     |
| Telefone:[_________________]     |
| [ Adicionar contato ]            |
+----------------------------------+
| Pedro Silva                      |
|   pedro@email.com — (11) 9999   |
| ------                           |
| Ana Costa                        |
|   ana@email.com — (21) 8888     |
+----------------------------------+
```

**Requisitos funcionais:**

1. Formulário com 3 campos (nome, email, telefone), cada um com a validação visual do ex06.
2. Validação:
   - Nome ≥ 2 caracteres
   - Email contém `@` e `.`
   - Telefone tem pelo menos 8 dígitos (use `.replace(/\D/g, '')` pra tirar tudo que não é dígito antes de contar)
3. Ao enviar com validação OK:
   - Monta `titulo = nome + ' | ' + email + ' | ' + telefone`.
   - Faz POST na API com `aluno_id` e esse título.
4. **Lista de contatos** atualiza sozinha após cada cadastro.
5. **Separe** os 3 campos na renderização — para cada tarefa, faça `var partes = t.titulo.split(' | ');` e mostre `partes[0]` em destaque, `partes[1]` e `partes[2]` em linha menor.
6. Cabeçalho da lista mostra "Agenda — N contatos".

**Requisitos de organização:**

7. Funções separadas: `validar()`, `enviar()`, `carregarLista()`, `renderizarContato(tarefa)`.
8. Constantes no topo: `API`, `ALUNO_ID`.

**Bônus:**

9. Se a tarefa não tiver 3 partes ao fazer `.split` (ex: alguma tarefa antiga sem esse formato), mostre "Contato com formato inválido — id N" em vermelho. Não quebre.
10. Botão **"× Apagar"** ao lado de cada contato. (Aviso: a Aula 5 vai cobrir DELETE direito. Por enquanto, você pode esconder do DOM ao clicar, mesmo sem chamar o servidor — vai ressurgir no próximo F5.)
